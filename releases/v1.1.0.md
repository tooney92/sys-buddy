# sys-buddy v1.1.0 — 2026-07-24

Minor release. One large feature (**todos**), two smaller ones, and the governance
docs. No breaking changes to the tool surface; one **agent-visible behaviour change**
that is worth reading before you upgrade (see *The task's state became a rollup*).

Full write-up with screenshots: [`features/v1.1.0/`](../features/v1.1.0/README.md).

---

## Added

### Todos — several deliverables under one task

Before this release, one task meant one contract. Real work does not arrive that way:
you agree on "sign-in", then discover it is really four things, and there was nowhere
to put the other three.

A task can now carry several **todos**, each with its own contract chain and its own
`proposed → locked → built → verified` march. Todos surface *from the conversation* —
by the time you know them, the host setup screen is long gone — so they are
**agent-proposed and peer-accepted**, with no human approval gate. That is not new
authority: `propose_contract` / `lock_contract` never had a human sign-off either.
Control comes from the prompt's "propose only when your human directs it", and todos
follow the identical rule.

New tools: `propose_todo`, `accept_todo`, `decline_todo`, `repropose_todo`,
`drop_todo`, `get_todos`. `propose_contract`, `lock_contract` and `report_status` gain
a todo selector.

- **Proposing IS the creator's consent.** You must be one of the named parties — you
  cannot propose work that binds only other people.
- **Accepting agrees on WHAT.** The contract on that todo is a separate, later
  agreement about HOW, signed by the same parties.
- **Declining carries a reason** and is recorded as a list entry beside the
  acceptances, not as a `declined` status. A status would have to be unwound the moment
  the creator reproposes, and it would lose "who said no, and why".
- **`repropose_todo` issues a new version and resets every acceptance**, so nobody is
  held to a scope they never read. Changing the party list is allowed precisely because
  everyone re-accepts afterwards.
- **Dropping is mutual** — every named party consents — with a host override through
  `sys-buddy todo drop <task> <todo> --reason "…"`.

#### Seats are not participants

A todo reuses the task's existing seats and names *which of them it binds*. You pair
once, at the task. A seat that is not a party on a todo may still READ it (it may need
to know the work exists), but it is not bound by it, is not in that contract's quorum,
and does not block it.

#### The task's state became a rollup

**This is the agent-visible change.** A task's state is no longer something an agent
sets — it is DERIVED from its todos, and agents stop calling `_transition` on the task.

The cost is real: the task state machine stops being agent-driven. It is worth paying,
because a rollup *cannot disagree with its parts* the way an agent-set task state can
drift from them. "The task concludes when the last todo verifies" then falls out
instead of needing a special case.

Two deliberate refusals in the rollup:

- `verified` requires **all** live todos. Any other state is the furthest any single
  live todo has reached.
- `stuck` is **never** derived. A stuck todo is surfaced as a count so a human sees it,
  but it must not force the task into a terminal state that would freeze the other five
  deliverables. On a todo, `stuck` is an orthogonal flag, not a state.

### The dashboard: the stepper truncates, it does not die

Once todos exist, `open` and `pre-flight assessments` remain task-level (you pair once,
at the task); planning / locked / backend live / testing / verified all fan out per
todo. So the task stepper keeps its head and its tail becomes the fan-out — the same
component, the same visual language, **three nodes** carrying a progress bar,
`2 of 6 verified`, and `⚠ 1 awaiting acceptance`.

The five dropped phases move INTO each todo as its own mini-stepper
(planning → locked → build → verified), so nobody learns a second idiom.

The right column becomes the **todo list**. Pending todos sort to the TOP with a `⚠` —
a todo awaiting your acceptance is the only thing on that screen blocking a human;
everything else is progress, that one is a request. Selecting a todo swaps the panel to
the same contract card, scoped to it.

**One thread, not six.** The task keeps a single message thread, with a
`⟨#3 Login endpoint⟩` chip on messages belonging to a deliverable. Six threads would
fragment a conversation that is genuinely one conversation — you would lose "we
discussed refunds while building payments".

The task-list row gains the same rollup (`2/6 verified ⚠1`) so a host can triage
without opening anything.

**The host's drop is read-only.** Per decision D11 the dashboard never issues commands:
the `Drop todo (host)` affordance prints the exact CLI line to type in your own
terminal. No button, no write path, host viewers only — a leaked `?v=` link must only
ever be able to look.

**Tasks with no todos are untouched.** The API omits the todo keys entirely, so a
pre-todo payload is byte-identical to 1.0.1. Verified by diffing the dashboard's
rendered DOM before and after the change: a debug task is byte-for-byte identical, and
a no-todo contract task differs only by one inert HTML attribute. A task whose todos
were *all dropped* also falls back to the full seven-node stepper.

### Always-listening presence

An agent parked in `wait_for_message` now shows a live pulsing dot and a
`listening — 42m` streak, so you can see at a glance that a peer is genuinely waiting
on mail rather than gone.

Presence is stored as an **expiry, not a boolean**. A boolean would persist a lie if
the broker died with agents parked — the clearing `finally` never runs, and the rows
would claim "listening" forever. Every wait is bounded by a cap, so
`listening_until > now` is self-healing with no cleanup job.

It is a **live-only** signal: it shows only while an agent is actually parked, and goes
out the moment the wait returns.

## Changed

- **Contract lock is pushed, not polled.** When a contract locks, the broker authors a
  `contract_locked` notification to the roles that already signed.
- **`staging_url` is collected at host setup**, with validation strictness keyed to the
  task's connectivity and `localhost` permitted for same-machine tasks.
- **`messages.todo_id` is stored at post time.** The dashboard's `⟨todo⟩` chip reads the
  column instead of scraping "todo #N" out of the message body, which silently missed
  any todo-scoped message whose text did not happen to name the deliverable. The regex
  survives only as a fallback for rows written before the column existed, and either
  candidate is validated against the task's real todos, so a stale `#999` gets no chip.

## Migration

No action required. On broker start, `db.init_db` adds the `todos`, `todo_decisions`
and `todo_drop_consents` tables plus the `contracts.todo_id` and `messages.todo_id`
columns, `NULL` on every existing row. Every existing task keeps working with no todos
at all — the migration story is "do nothing".

**Restart any long-running broker.** A broker process started before this release keeps
serving its old code against an unmigrated database, so none of the above will appear
no matter what the agents do.

## Tests
