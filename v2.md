## Contract lock notification (push instead of poll)
**Context:** `lock_contract` returns `{locked: true}` synchronously to whoever signs LAST,
but the FIRST signer gets no push when the lock later completes. Today (v1) they must poll
`get_contract()` until `status == "locked"` — documented in the agent prompt.

**v2:** make the lock *pushable* — when the final signature lands, the broker drops a
synthetic event/message to the already-signed role(s) so `wait_for_message` wakes on it
(a `contract_locked` lifecycle event, or a broker-authored message). No polling; cleaner
agent ergonomics. Touches: `state.lock_contract` (emit to signed roles), `service`/`wait_for_message`
(surface lifecycle events), prompt (drop the "poll `get_contract`" line).

## Stronger auth: mTLS / OAuth 2.1 (Tier-3 hardening)
**Context:** v1 identity is bearer-token based ("whoever holds the token is in"). Mitigated by
scoped tokens, 24h TTL in tunnel mode, `rotate_token`, single-use invites, rate limiting, and
`close_task` revocation — but there is no cryptographic proof-of-identity, so a leaked token is
usable until it expires or is rotated/revoked.

**v2:** add real proof-of-identity so a stolen token alone isn't enough:
- **mTLS** — client certificates per agent; the broker verifies the cert on every request.
- **OAuth 2.1** — proper authorization-code / device flow with short-lived access tokens +
  refresh, instead of long-lived bearer tokens.
Removes the bearer-holder risk (#2 in the security review) and complements the Tailscale path,
which already removes the ngrok-terminates-TLS middleman risk (#1). Touches: `identity`/`middleware`
(auth verification), `pairing` (cert/OAuth issuance), config, docs.
