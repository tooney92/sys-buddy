## Architecture / key facts
- New modules: `onboarding.py` (pure engine: invite-link `sb1_` encode/decode, `role_prompt`, `claude_add_command`/`configure_claude` (never raises), `pair`, `join_flow`, `host_create_task`/`host_invite_link`/`host_setup`). `gui.py` (pywebview shell + `GuiApi` bridge + in-app broker thread). `gui_app.html` (single-page app: home/host/buddy/done + `showView` router). `tests/test_onboarding.py`.
- The GUI runs the broker **in-process on a daemon thread in REMOTE mode** — required so buddy agents auth by bearer token and get the parameter-free tool surface (local mode's tools take task/agent params → wrong for the MCP buddy). Dies with the app.
- CLI: `sys-buddy gui` (lazy-imports gui so the CLI never needs pywebview). Dep: `pywebview>=5.0`.
- The GuiApi bridge NEVER raises across to JS — every method returns `{...}` or `{"error": str}`.
